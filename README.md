# bsky-hashtags-trend-poster

## これは何？

Bluesky Jetstreamを使い、指定した言語のポストに含まれるハッシュタグを集計しポストするためのGoプログラムです。docker composeでbuild、実行するためのファイルも含めてあります。

## 使い方

### ビルドと起動

docker composeを使ってビルドから起動まで行えます。詳細は各設定ファイルを読めば明らかですが、defaultではrepositoryをcheckoutしてきたdirectoryに `work` というdirectoryを作成し、以下のような内容を持つ `env.txt` ファイルを作成して同じ場所に置けばすぐ起動させられるものと思います。指定する環境変数は、 `BSKY_ID` がポスト時に利用するBlueskyのID、 `BSKY_APPPASSWORD` がそのアカウントで生成したアプリパスワードで、 `TARGET_LANG` には集計したいポストの言語を指定してください。

```
BSKY_ID=hashtags-trend-ja.bsky.social
BSKY_APPPASSWORD=xxxx-xxxx-xxxx-xxxx
TARGET_LANG=ja
```

その後、いつものように下記のような感じでビルド、起動出来ます。あ、 `Dockerfile` 内で生成しているcontainer内user、groupのIDは、実行される環境に合わせて適宜修正する必要があります。

```
(repositoryをcheckoutしたdirectoryで)

$ docker compose build # ビルド

$ docker compose up -d # 起動

$ docker compose down  # 終了

$ docker system prune  # ビルド時に生成されるゴミの掃除
```

## あれこれ

- ポストに含まれるハッシュタグの抽出は、Jetstreamに流れてくる `FeedPost` データの含まれる `Facets` データから行っています。
  - ちなみに集計結果をポストする際にもポストに含まれる個々のハッシュタグに対応する `RichtextFacet_Tag` データを付ける必要があるのですが(そうしないとタグがリンクにならない)、この時に指定出来るタグの長さには64bytesという制約があるのに、Jetstreamに流れてくるポストに含まれるfacet中のタグにはもっと長いものが含まれることがあり、謎です。
  - 当初facetからではなく本文から正規表現でハッシュタグを抜き出したりしていましたが、それだとbsky側ではハッシュタグと認識されていないものも含まれることがあり、抽出方法を見直しました。なお正規表現時代には抽出したタグにUnicodeのZero-width charactersやVariation selectorが含まれていることがあって、それがまた集計の邪魔をしたりしたことがあったので、(facetから取るようにしたので不要になったのかもしれないけれど)その辺りをtrimするコードは念のためそのまま残しています。

## AIについて

本ツールの作成にあたり、極力生成AIは利用しないよう努めました。

去年の中頃までは僕も世のソフトウェアエンジニアの例に漏れず、生成AIによるコーディング支援を積極的に利用していました。ただその後、学習データの権利侵害や利用規約違反について声を上げている著作者の方々がおられることを知り、いろいろ自分なりに調べ考えてみて、どうしても僕にはそういった人達の声を無視することはできない、現行の法律では訴えたくても訴えられない、被害の立証が極めて困難な非対称性、言ってしまえば搾取構造があるとしか思えない、と感じたため、プライベートでは学習元のはっきりした(=透明性のある)生成AI以外は使わないと決めました。

> 現在AIに対する法規制が最も進んでいるのはEUだと思いますが、そのEUのAI Actでも、学習データの透明性に関しては大手生成AIベンダーの強い反発(曰く、業務機密の開示に当たるから、とのこと)を受けて詳細な開示までは踏み込めず、学習元データの概要の開示までに留まっています。(2026年5月現在)

そんなわけで本ツールの作成にあたって生成AIは全く利用しませんでしたが、とはいえ自分が過去生成AIを利用しつつ書いたコードは参照していますし(多くの部分は今回手で書き換えていますが)、Webで情報を探す際に検索エンジンが返すAIによる要約もどうしても目にしてしまいますので(少し前からGoogleではなくDuckDuckGoを利用していますが、そちらでもdefaultではAI要約が表示されます。つい先日offに出来ることを知ったので今後はoffで使うつもり)、完全にAI freeなコードとは言えません。その点を気にされる方はご留意いただければと。

なおGolangを使って自分で本格的なcodeを書くのは初めてでしたが、仕様がコンパクトでとても書きやすかったです。去年、生成AIにGo codeを書いてもらった時はうまく動くようになるまでかなり苦労した記憶があるのですが(bsky indigoライブラリのように更新頻度の高いライブラリだとLLMが持つ知識では古くなっていて正しいcodeが生成出来ないことがままある)、今回自分で情報を調べて書いてみて、僕の場合はこっちの方が圧倒的に早く、確実なcodeが書けました。僕の場合は効率という意味でも精神衛生上の意味でも、生成AIを使わずにcode書く方が良いようですｗ。
